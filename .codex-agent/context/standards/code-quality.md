# Code Quality

- Make the smallest change that fully satisfies the approved scope.
- Follow nearby naming, architecture, error handling, and test patterns before introducing a new abstraction.
- Keep public behavior explicit and preserve compatibility unless the task authorizes a breaking change.
- Prefer cohesive modules and clear data flow over speculative generalization.
- Remove placeholders, dead code, debug output, and temporary artifacts before completion.
- Explain unavoidable tradeoffs and leave concrete follow-up work rather than hidden debt.
